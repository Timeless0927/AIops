# PDD：产品定义

## 产品目标

面向 Kubernetes 告警修复场景，AIOps Agent 在飞书协作环境中完成告警接入、诊断摘要、人工审批和受控执行。高风险或会改变生产状态的动作必须先获得人类批准，但执行权威始终属于 AIOps：飞书只决定“人类是否批准”，不直接触发命令执行。

CR-2026-05-15-001 后，飞书原生审批成为主人工审批入口。AIOps 创建本地 approval 后发起飞书原生审批实例，飞书审批中心承载审批人、审批链和审批记录；AIOps 本地审批状态机、执行安全、operation lock、审计、健康检查和回滚仍是系统事实源。

## 目标用户

- 值班 SRE：在飞书中接收告警、查看风险摘要、完成审批，并关注执行结果。
- 平台运维负责人：定义哪些修复动作必须审批，关注审批闭环、审计证据和失败可恢复性。
- 审计/安全审查人员：追踪“谁批准了什么、何时批准、AIOps 是否按批准范围执行”。

## 要解决的问题

- 告警修复建议需要在人类批准后才能执行，避免高风险动作被自动化系统直接落地。
- 审批入口必须符合企业协作习惯，复用飞书原生审批的审批人权限、审批链和审批记录。
- 外部审批事件可能丢失、重复或延迟，AIOps 必须保留本地状态机，确保只有本地 `approved` 才能进入执行。
- 自定义审批卡片容易把“通知已发送”误解为“审批事实已成立”。CR-2026-05-15-001 后，自定义审批卡片只作为通知展示/回退，不再作为主审批事实。

## 当前范围

- Alertmanager 告警触发 Incident 创建、诊断摘要和修复建议生成。
- 对需要审批的修复动作，AIOps 创建本地 approval，并发起飞书原生审批实例。
- 飞书审批事件 webhook 或 polling 补偿把外部审批结果同步为本地状态：`approved`、`denied`、`canceled` 或 `expired`。
- 飞书审批创建失败时，本地状态进入 `approval_create_failed`，动作不得执行。
- incident thread 展示审批已发起、审批链接、风险摘要和操作摘要；自定义审批卡片仅作为通知展示或回退入口。
- 只有 AIOps 本地 approval 状态进入 `approved`，execution worker 才能在安全门禁和 operation lock 约束下继续执行。

## 明确不做

- 不让飞书审批 webhook、飞书卡片按钮或任何外部回调直接执行修复命令。
- 不把飞书作为 AIOps 的唯一事实源；飞书审批结果必须先同步到本地 approval。
- 不再把自定义审批卡片作为主审批入口或主审批事实。
- 不在产品语义上绕过本地安全门禁、审计、operation lock、健康检查和回滚。
- 不要求 SRE 在飞书外单独维护审批链；审批人权限和审批链由飞书原生审批承载。

## 成功指标

- 需要审批的修复动作 100% 先产生本地 approval，再发起飞书原生审批实例或进入明确失败状态。
- `external_pending`、`approval_create_failed`、未知外部事件和非法回调均不会触发执行。
- 飞书 `APPROVED` 只会把本地状态同步为 `approved`；真正执行必须由 AIOps execution worker 读取本地状态后触发。
- `denied`、`canceled`、`expired`、`executed`、`failed` 等终态不会被重复、延迟或 polling 事件回滚。
- incident thread 中能看到审批链接、风险摘要和操作摘要，且不会把自定义卡片展示误导为主审批入口。

## 约束

| 约束 | 影响 | 验证方式 |
| --- | --- | --- |
| 飞书只表达“人类是否批准” | 外部事件不得绕过本地状态机直接执行 | BDD 场景覆盖 webhook 同步后仍需本地 execution worker 执行 |
| AIOps 是执行权威和事实源 | 所有执行判断必须基于本地 approval、operation lock 和安全门禁 | DDD 不变量要求只有本地 `approved` 可执行 |
| 飞书原生审批是主入口 | 自定义审批卡片降级为通知展示/回退 | 验收检查 thread 展示不被视为审批事实 |
| 外部事件可能丢失、重复或延迟 | 需要 polling 补偿、幂等同步和终态保护 | BDD 覆盖 webhook 丢失、重复事件和终态不可回滚 |

## 风险

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 飞书审批实例创建失败 | SRE 可能以为审批已发起，实际无法审批 | 本地进入 `approval_create_failed`，thread 明确失败，不执行动作 |
| 飞书 webhook 丢失或延迟 | `external_pending` 长时间无法收敛 | polling worker 补偿同步，并保留失败退避和审计 |
| 重复或乱序事件覆盖终态 | 已拒绝、取消、过期或已执行的审批被错误改回可执行 | 本地状态机拒绝终态回滚，事件只记录审计 |
| 自定义卡片语义残留 | SRE 误用旧卡片作为审批入口 | 文档和验收明确卡片仅通知/回退，主入口为飞书原生审批 |
